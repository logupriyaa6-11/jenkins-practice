## What is terraform?
- Terraform is an IaC (Infrastructure as Code) tool by hashicorp
- It lets us create, update and manage infrastructure such as AWS, Azure using simple configuration files instead of clicking the buttons in cloud console

## Terraform Core Concepts
Terraform works around a few core ideas:
    Configuration files (.tf): Describe what infrastructure should exist
    Providers: Plugins that interact with APIs (AWS, Azure, GCP, etc.)
    State: Tracks what Terraform has already created
    Execution plan: Shows what Terraform will change before doing it

## Uses of terraform
- Infrastructure as Code (Servers, networks and services are written down in the form of code)
- Human readable files (The files are easy to read and understand)
- version control friendly (These files can be stored in Git so that we can track the changes and rollback if needed)
- works everywhere (It is used to manage cloud services and on-prem servers-servers that we own)
- Same process everytime (one workflow to create, change and destroy the infrastructure)
- Handles small and big things (It can handle from basic things like servers and disk to higher level things like DNS records and Saas settings)

## Workflow
1. Write - Defining an infrastructure in config files
2. Plan - creates a plan describing the infrastructure
3. Apply - performs proposed operations in the correct order

write -> terraform reads -> terraform creates/chnages/deletes infrastructure 

## Provider (plugin)
- Terraform works by intracting with the cloud services by using APIs
- To intract with the cloud we use providers
- Provider is like a translator between Terraform and a service 
- already exist, so no need of building anything by ourselves
Examples:
    AWS provider - manages AWS resources
    AzureRM provider - manages Azure resources
    Google provider - manages GCP resources

## Terraform State
Terraform maintains a state file (terraform.tfstate) that tracks:
    What resources Terraform manages
    Resource IDs and metadata
    Relationships between resources

Terraform uses this state to determine:
    What already exists
    What needs to be changed
    What should be destroyed

## Local State vs Remote state
local state:
Stored on disk (terraform.tfstate)
Simple, but not safe for teams

Remote state:
Stored in a remote backend (S3, Azure Blob, Terraform Cloud)
Enables collaboration
Supports state locking

## Terraform Lock File (.terraform.lock.hcl)
The .terraform.lock.hcl file records exact provider versions used by your project. It ensures that:
Everyone uses the same provider versions
Builds are reproducible
Provider upgrades are controlled

Why it is important? 
Automatically generated by terraform init command
Should be committed to version control
Prevents unexpected provider version changes

## Terraform commands:
1. terraform init
Initializes a Terraform project

What it does:
Downloads providers
Sets up backend
Creates .terraform/ directory and .lock.hcl file

2. terraform plan
Creates an execution plan showing what Terraform will do
Outputs:
Resources to be created (+)
Resources to be modified (~)
Resources to be destroyed (-)

3. terraform apply
Applies the planned changes to real infrastructure
Behavior:
Shows a plan
Prompts for confirmation
Executes changes if approved

4. terraform destroy
Destroys all infrastructure managed by the configuration
Important notes:
This is destructive
Requires confirmation
Uses state to determine what to delete

5. terraform validate
Checks configuration files for syntax and internal consistency
What it does:
Catch syntax error
Validate references
Integrate into CI pipelines

6. terraform fmt
Formats Terraform files to standard style
